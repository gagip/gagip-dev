## 입력

```
$ARGUMENTS
```

## 핵심 원칙 (항상 이 원칙을 기준으로 판단한다)

이 스킬은 **수직 슬라이스 아키텍처(VSA)** 를 따른다.

**세 가지 핵심 가치:**
- **변경의 지역성** — 기능을 수정할 때 해당 슬라이스만 건드린다
- **슬라이스 내 응집도 극대화** — 관련 코드가 한 곳에 모인다
- **슬라이스 간 결합도 최소화** — 슬라이스는 서로를 직접 참조하지 않는다

**기본 구조:**
```
app/
  core/           — 앱 전반 공유 인프라
  features/
    {feature}/    — 기능별 수직 슬라이스 (내부 패키지는 자유)
```

**의존 흐름 (단방향):**
```
feature → core
feature ✗→ feature  (슬라이스 간 직접 참조 금지)
```

**슬라이스 간 통신 방법:**
1. 원시 타입 전달 — 도메인 모델 대신 ID 등 원시 타입 사용
2. 이벤트 기반 — `core/events/`에 이벤트 모델(원시 타입만 포함)을 두고 발행/구독

**core 기준:**
- 넣는 것: 공유 인프라, 앱 전역 유틸리티, 이벤트 모델, 반복이 확인된 공유 코드
- 넣지 않는 것: 특정 슬라이스의 비즈니스 로직, 특정 슬라이스에서만 쓰는 모델

**슬라이스 설계 원칙:**
- 다른 슬라이스와 코드가 비슷해도 **일단 슬라이스 내부에 둔다**
- 중복이 반복되고 공유 필요성이 명확해졌을 때만 core로 올린다

## Step 1 — 모드 결정

다음 기준으로 모드를 자동 판단한다.

| 상황 | 모드 |
|------|------|
| 기존 파일/폴더/코드가 주어졌거나, "봐줘", "리뷰", "어때", "맞아?" 등 검토 요청 | **피드백 모드** |
| 새 기능·시스템 설명이 주어졌거나, "설계", "구조 잡아줘", "어떻게 나눠" 등 설계 요청 | **설계 모드** |
| 둘 다 섞임 | 피드백 먼저 → 개선 설계 제안 |

판단 후 진행 전에 모드를 한 줄로 명시한다. 예: `[피드백 모드]` 또는 `[설계 모드]`

## Step 2A — 피드백 모드

### 대상 파악

- `$ARGUMENTS`가 있으면 해당 경로를 Read/Glob으로 읽는다
- 없으면 현재 디렉토리에서 `features/`, `core/` 구조를 탐색한다
- 전체 구조 파악이 필요하면 폴더 트리를 먼저 확인한다

### 체크리스트

다음 항목을 순서대로 확인한다. 해당 없는 항목은 생략한다.

**슬라이스 경계**
- 기능이 명확한 슬라이스로 분리되어 있는가
- 슬라이스가 다른 슬라이스를 직접 참조하는가 (import, 의존성 확인)
- 슬라이스 간 전달값이 원시 타입인가, 도메인 모델인가

**core 기준 준수**
- core에 특정 슬라이스 전용 코드가 섞여 있는가
- 슬라이스에 있어야 할 코드가 core에 있는가
- core에 있어야 할 인프라 코드가 슬라이스에 흩어져 있는가

**응집도**
- 하나의 기능과 관련된 코드가 한 슬라이스에 모여 있는가
- 관련 없는 코드가 같은 슬라이스에 묶여 있는가

**변경의 지역성**
- 이 구조에서 특정 기능을 수정할 때 여러 슬라이스를 건드려야 하는가

### 출력 형식

```
## 아키텍처 피드백

### 📋 분석 대상
(파일/폴더 목록)

### ✅ 잘된 점

### ⚠️ 개선 사항
- 🔴 Critical — 즉시 수정 필요 (슬라이스 간 직접 참조 등 핵심 원칙 위반)
- 🟡 Warning — 수정 권장
- 🟢 Suggestion — 개선 고려

각 항목: 문제 설명 + 위치(파일경로 또는 패키지) + 개선 방법

### 📝 총평
```

## Step 2B — 설계 모드

### 요구사항 파악

설계 전에 다음이 불명확하면 질문한다 (한 번에 묶어서, 과도하게 묻지 않는다):
- 어떤 기능인가
- 이미 존재하는 슬라이스와 연관이 있는가
- 플랫폼/언어 (프로젝트 컨텍스트로 파악 가능하면 생략)

### 설계 원칙

설계 시 이 순서로 판단한다:

1. **슬라이스 분리** — 이 기능이 독립 슬라이스인가, 기존 슬라이스 확장인가
2. **내부 구조** — 슬라이스 내 패키지를 어떻게 나눌지 (플랫폼과 복잡도에 맞게 자유롭게)
3. **다른 슬라이스와의 접점** — 원시 타입 전달인가, 이벤트인가
4. **core 후보** — 공유가 필요한 게 있는가, 아니면 일단 슬라이스에 두는가

### 출력 형식

```
## 아키텍처 설계

### 🗂️ 슬라이스 구조
(제안 폴더 구조 — 코드 블록)

### 📦 각 패키지 역할
(패키지별 책임 설명)

### 🔗 슬라이스 간 통신
(다른 슬라이스와의 접점이 있으면 통신 방법 명시)

### 🏗️ core 후보
(core에 올릴 것이 있으면 명시, 없으면 생략)

### 💡 설계 결정 근거
(왜 이렇게 나눴는지 핵심 이유)
```

## 행동 원칙

- 개선 방법 제시 시 구체적인 예시(코드 또는 구조)를 포함한다
- 원칙 위반을 지적할 때는 **왜** 문제인지 설명한다 (변경의 지역성, 결합도 등 구체적 근거)
- 불확실한 것은 추측하지 않고 질문한다
- 피드백 후 수정은 하지 않는다 — 제안만 한다 (사용자가 명시적으로 수정 요청 시 제외)
