## 입력

```
$ARGUMENTS
```

`$ARGUMENTS`가 있으면 해당 task-name으로, 없으면 현재 대화 맥락에서 task-name을 추론. 맥락에서 task-name을 특정하기 어려우면 사용자에게 확인 후 진행.

---

## 작업 순서

### 1. 정보 수집

다음 소스를 우선순위 순으로 활용:

1. **현재 대화 내역** — 대화에서 진행한 작업, 결정, 논의 내용
2. **`dev/active/<task-name>/`** — plan, tasks, context 파일 (존재하는 경우에만)
3. **변경된 파일** — git diff 또는 직접 읽기로 실제 변경 내용 확인

`dev/active/`가 없어도 대화 내역만으로 보고서 작성 가능.

### 2. 보고서 작성

`dev/report/<task-name>-YYYY-MM-DD.md` 파일 생성 (날짜는 보고서를 작성하는 오늘 날짜 기준):

```markdown
# <task-name> 작업 보고서

## 작업 배경
- 해결하려 한 문제 또는 필요성
- 작업 범위와 제약 조건

## 접근 방식
- 선택한 방법과 그 이유
- 검토했으나 선택하지 않은 대안 및 이유 (있을 경우)

## 주요 결정 내역 (결정 이유가 있는 경우에만, 없으면 섹션 생략)
| 결정 항목 | 배경 | 선택 | 이유 |
|-----------|------|------|------|
| ...       | ...  | ...  | ...  |

## 구현 요약
- (변경 파일 경로 및 핵심 함수명만)

## 이슈 및 해결
- (발생한 문제, 원인, 해결 방법 — 없으면 생략)

## 남은 과제
- (후속 작업, P2/P3 항목 — 없으면 생략)
```

### 3. active 정리 (해당하는 경우에만)

`dev/active/<task-name>/`이 존재하는 경우, 보고서 작성 완료 후 사용자에게 확인:

```
보고서를 작성했습니다: dev/report/<task-name>-YYYY-MM-DD.md

dev/active/<task-name>/ 폴더를 삭제할까요?
```

- 승인 시 `dev/active/<task-name>/` 삭제
- 거절 시 그대로 유지
- `dev/active/`가 없었다면 이 단계 생략

### 4. 완료 알림

```
완료: dev/report/<task-name>-YYYY-MM-DD.md
```

## 행동 원칙

- 보고서는 간결하게, 섹션당 5줄 이내
- 코드 발췌 금지 (파일 경로와 함수명 언급으로 대체)
- "무엇을 했는가" 보다 "왜 이렇게 결정했는가"에 집중
- 결정 이유가 없는 결정 항목은 기록 금지
- 개발자 관점에서 "이 코드를 왜 이렇게 짰는가"에 답할 수 있어야 함
