## AI 협업

> 근본 원칙의 **실수할 여지를 없앤다** 원칙을 AI 협업 레벨로 확장한다.

### 핵심 인식

- AI는 확률 모델이다. 코드는 결정론적이어야 한다
- AI가 생성한 코드를 결정론적 범위 안으로 제한할 필요가 있다
- 이를 제한하는 장치가 **하네스**
- 계약 검증이 코드의 올바름을 보장하듯, 하네스가 AI 출력의 올바름을 보장한다

### 역할 분담

AI는 보조 인력이다. 각자 잘하는 영역이 있다.

| 영역 | 사람 | AI |
| --- | --- | --- |
| 목적 정의 | 왜 만드는가, 무엇이 중요한가 | — |
| 판단 | 트레이드오프 결정, 아키텍처 선택 | 선택지 제공 |
| 맥락 | 프로젝트 방향성, 사용자 관점 | 문서에서 참조 |
| 탐색 | 탐색 방향 설정 | 코드 검색, 지식 조회, 다른 플랫폼 사례 |
| 실행 | — | 코드 생성, 반복 작업, 보일러플레이트 |

**모호한 케이스의 원칙: AI가 선택지를 제공하고, 대화를 통해 함께 구체화하며, 최종 결정은 사람이 한다.**

### 검증

핵심 로직은 직접 확인, 나머지는 테스트에 위임한다. 책임은 항상 사람에게 있다.

| 코드 종류 | 처리 |
| --- | --- |
| 보일러플레이트/fixture | 그대로 커밋 가능 |
| 핵심 비즈니스 로직 | 사람이 의도대로 동작하는지 확인한 후 커밋 |
| 테스트 코드 | 사람이 의도대로 동작하는지 확인한 후 커밋 |

### 하네스 체계

**필수** — AI 이전부터 존재하는 보편적 검증 수단

- 테스트 코드: AI 출력의 올바름을 검증
- CI: format, lint, build, test 자동 실행
- 코드 리뷰: AI 코드를 사람이 확인하는 최종 관문
- 아키텍처 룰: 정적 분석 커스텀 룰로 구조 위반 자동 감지
- 계약 검증: 런타임에서 사전조건/사후조건/불변식 위반을 잡는 코드 내 가드레일

**보강** — AI 협업을 강화하는 추가 장치

- 스킬: AI의 행동 패턴을 사전에 정의
- 훅: 특정 시점에 자동으로 검증 실행
- CLAUDE.md / 문서: AI에게 프로젝트 컨텍스트와 원칙을 전달하는 계약

### 계획

**구현의 질은 계획의 질에 비례한다.** 사람이 가장 집중해야 할 단계다.

- 계획이 명확해야 대화가 명확하고, 대화가 명확해야 AI 출력이 명확하다
- 문서는 기본 맥락, 대화는 실시간 보충 — 둘 다 필요
- 모호한 부분이 남아 있으면 AI는 진행하지 않고 물어봐야 한다

### 책임 소재

- AI가 실행하고, 사람이 확인하는 루프가 필수적이다
- AI가 수행한 작업은 사람이 확인할 수 있는 형태로 기록되어야 한다 — 기록이 없으면 판단할 수 없다
- 사람이 기록을 확인하고 최종 결정을 내린다
