## 아키텍처

> 근본 원칙의 **심플함**과 **불변성** 원칙을 아키텍처 레벨에서 적용한다.

### 범용 원칙

**VSA (Vertical Slice Architecture)**

- 슬라이스끼리 구분, 공유는 shared/core로 — 원칙이 심플하다
- 레이어 아키텍처는 레이어 분리로 인한 부수적인 코드(mapper, interface 등)가 번거롭다
- VSA는 기능 단위로 바로 작업할 수 있다 — 기능과 코드가 1:1로 대응
- 슬라이스 내부는 좁은 접근 제어로 경계를 강제한다

**단방향 상태 관리**

- 데이터 원천이 하나 (단일 상태) — 외부 변경을 차단
- 상태 기반 로직은 직관적이다
- 단일 상태이므로 버그 추적이 쉽다
- 에러도 상태로 표현한다 (대수적 타입)

**슬라이스 간 경계**

- 공유 코드: 실제 공유가 발생할 때만 shared로 올린다 — 예측이 아니라 현실 기반
- shared 모델은 순수하게 유지, 슬라이스별 관심사는 슬라이스가 자기 표현 모델을 정의
- 슬라이스 간 통신: 직접 의존이 아닌 간접 통신 (시그널/이벤트 버스 패턴)
- 라우트 같은 글루 코드는 shared에 계약으로 모아 한 곳에서 관리

**슬라이스 내부 구조**

- 내부에서만 쓰이면 파일 안에 응집
- 외부에서도 참조하면 별도 파일로 분리
- "단일 상태"는 "하나의 클래스"가 아니라 "하나의 역할에 하나의 상태"
- Side Effect는 최대한 상태로 표현한다 (에러, 네비게이션 등)

**계층별 역할**

| 계층 | 역할 |
| --- | --- |
| Repository | 데이터 접근 계약 |
| Manager | 하나의 도메인 개념에 대한 비즈니스 로직 응집 |
| UseCase | 여러 소스를 조합하는 단일 행동이 슬라이스 간 재사용될 때만 |
| 유틸 함수 | 순수 변환, 부수효과 없음 |

**에러 처리**

Rust의 에러 처리 원칙에서 영감: **에러는 명시적이어야 하고, 호출자의 필요에 따라 상세도를 결정한다.**

핵심 질문: "호출자가 이 에러의 종류에 따라 다르게 행동하는가?"

| 구분해야 하는가? | 선택 | 예시 |
| --- | --- | --- |
| No — 로깅/표시만 | 공통 에러 | 네트워크 실패 → 메시지 표시 |
| Yes — 분기 처리 필요 | 슬라이스별 에러 | 리소스 해제 → 정지, 데이터 없음 → 다음 항목 |

계층별 책임:

- Repository: 저수준 예외를 잡아 도메인 에러로 변환
- Manager: 도메인 에러를 받아 상태로 표현
