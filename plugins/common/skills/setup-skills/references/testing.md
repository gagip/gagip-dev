## 테스트 전략

> 근본 원칙의 **명확성** 원칙을 검증 레벨에서 적용한다.

SQLite 개발자의 설계 중심 원칙, 단순함 원칙에서 영감을 받았다.

### 생각의 변화

- 원래 테스트를 잘 안 만드는 개발자였다
- 모킹 같은 번거로운 작업이 직관적이지 않았기 때문
- 수동 테스트의 한계, SQLite 개발자의 테스트 원칙, AI 시대의 안전 가드레일 필요성을 느끼며 생각이 바뀜

### 모킹

- 최소화한다. SQLite도 자체 모듈을 모킹하지 않고 실제 동작으로 검증한다
- 모킹보다 실제 환경에 가까운 테스트를 선호 — 실제 동작이 가장 명확한 증거
- 프레임워크 의존은 에뮬레이션/통합 테스트로 대체, 순수 로직은 모킹 없이 직접 테스트

### 목적 기반 테스트

단위/통합/e2e 같은 레벨이 아니라, **"이 테스트가 무엇을 증명하는가"**가 기준이다.

| 목적 | 방법 |
| --- | --- |
| 상태 전환이 올바른가 | 상태 관리 테스트 |
| 화면 흐름이 정상인가 | UI 테스트 |
| 슬라이스 간 연동이 되는가 | 통합 테스트 |
| 잘못된 입력에도 안전한가 | 경계값/에러 케이스 테스트 |

각 테스트가 증명하는 것이 다르므로 모두 필요하다.

### 우선순위

- **앱의 목적에 크게 부합하는 핵심 기능부터** 테스트한다
- "이게 깨지면 앱이 성립하지 않는가?" 순서로 우선순위를 정한다

### 작성 시점

- 구현 직후, 버그 발생 후, 안정화 후 — 모든 시점에서 작성한다
- TDD는 지양한다. 테스트를 위한 설계가 되면 안 된다. 설계는 목적에 맞게, 테스트는 설계를 검증하는 역할

### 컨벤션

**이름** — 행동/시나리오 기반, 한글로 작성. 테스트는 인간 중심에서 살펴본다.

- 예: `셔플 모드에서 다음 곡은 무작위 순서를 따른다`

**데이터** — 공통 fixture 사용. `TrackFixture.create()` 같은 팩토리로 중복 제거.

**독립성** — 각 테스트가 완전히 독립적으로 실행. 공유 상태에 의존하지 않는다.

**목적 기반 assertion** — 하나의 시나리오가 증명하려는 것을 모두 검증한다. assertion 개수가 아니라 "하나의 행동을 검증하는가"가 기준.

### 실패 대응

- 테스트가 실패하면 **코드를 먼저 확인**한다 — 테스트가 버그를 잡은 것이라고 가정

### 커버리지

- 커버리지는 참고 지표일 뿐 목표가 아니다
- 핵심 기능의 커버리지는 자연스럽게 높아야 하지만, 수치가 테스트의 완성도를 보장하지 않는다

### 사전 검증 — 스파이크

테스트가 *만든 코드*를 검증한다면, 스파이크는 *만들기 전 가정*을 검증한다. 둘 다 "추측 대신 실제 동작이 가장 명확한 증거"라는 같은 원칙의, 시점만 다른 적용이다.

- 구현이 **실험으로만 판정 가능한 미검증 전제**(외부 SDK 동작, 플랫폼 제약, 비동기 경로 등) 위에 서 있으면, 본 구현 전에 **버리는 최소 코드**로 그 한 점만 찔러 성공·실패를 먼저 가린다
- 묻거나 문서로 풀리는 불확실성은 스파이크가 아니다 — 그건 조사로 푼다. 스파이크는 **굴려봐야만 아는** 것에만 쓴다
- 통과 기준에 "**검증 대상 경로를 실제로 거쳤는가**"를 박는다 — 그 경로를 안 타고 떨어진 신호를 통과로 오판(false positive)하지 않도록
- **타임박스**를 두고, 그 안에 통과 신호를 못 보면 실패로 간주해 방향을 튼다
- 스파이크 코드는 버린다 — 검증이 끝나면 본 구현은 깨끗이 다시 짠다
