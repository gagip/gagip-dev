# VSA 공통 아키텍처

> 모든 플랫폼(Flutter, Android, Godot)에 적용되는 VSA 기본 원칙.
> 플랫폼별 세부 규칙은 `vsa-android-architecture.md`, `vsa-godot-architecture.md` 참조.

---

## 기본 기준 (우선순위)

| 우선순위 | 기준 | 설명 |
| --- | --- | --- |
| 1 | **단순함 = 독립성** | 독립성을 위한 복잡도는 허용. 그 외 복잡도는 제거 |
| 2 | **범용성** | 플랫폼마다 메커니즘이 달라도 됨. 억지 통일 금지 |

## 3단계 폴더 구조

```
app/        ← 글루 로직 전용 (home, router, di, 앱 뼈대)
features/   ← 순수 피처들 (독립적인 슬라이스)
shared/     ← 계약 + 최소 모델만
```

### 참조 방향 (단방향)

```
app/ → features/ 참조 가능
app/ → shared/   참조 가능
features/ → shared/ 참조 가능
features/ → features/ 직접 참조 금지
shared/ → features/ 참조 금지
```

---

## 슬라이스 단위

- **기본: 화면/씬 하나 = 슬라이스 하나**
- 분리 트리거 — 아래 중 하나가 발생하면 그때 나눈다
  1. 슬라이스가 너무 커졌을 때 (Flutter/Android: ~500줄, Godot: ~300줄)
  2. 두 기능이 서로 다른 이유로 변경될 때
  3. 같은 로직을 다른 슬라이스에서 복붙할 때 (두 번째 발생 시)

---

## 공유 코드 3단계

| 단계 | 위치 | 대상 |
| --- | --- | --- |
| 1단계 | 슬라이스 내부 | 이 슬라이스만 쓰는 것 |
| 2단계 | `features/{domain}/shared/` | 이 피처군의 슬라이스들만 쓰는 것 |
| 3단계 | `shared/` | 두 개 이상의 피처군이 쓰는 것 |

### 피처군 shared 생성 조건

- **처음엔 슬라이스 내부에 둔다**
- 같은 피처군의 두 번째 슬라이스가 동일한 코드를 필요로 하는 순간 `shared/` 생성 후 이동
- 미래를 예측해서 미리 생성 금지 (YAGNI)

### 피처군 shared에 넣는 것

✔ Repository, 도메인 Model, Exception, Enum/Const
✘ UseCase, API 응답 DTO (슬라이스 내부에 남음)
△ 공용 UI — 완전히 동일할 때만

### 전체 shared 기준

✔ 최소 공유 모델 (`*_summary`, `*_info`) — 필드 추가 금지
✔ 계약 인터페이스 — 구현체는 절대 올라오지 않음
✔ DI 연결 — 계약 ↔ 구현체 연결은 DI(app/)에서만
✔ 기술 인프라 — ApiClient, LocalStorage, Logger
✘ 구현체, 비즈니스 로직, 파생 모델

## 글루 로직

> 글루 로직은 `app/`에만 존재한다. `features/`와 `shared/`에 글루가 들어가면 냄새.

- `app/`은 여러 피처를 알아도 됨 (home, router, di)
- 글루는 연결만 담당 — 비즈니스 로직이 들어가면 해당 슬라이스로 이동

---

## 계층별 역할

| 계층 | 역할 |
| --- | --- |
| Repository | 데이터 접근 계약. 저수준 예외를 도메인 에러로 변환 |
| Manager | 하나의 도메인 개념에 대한 비즈니스 로직 응집 |
| UseCase | 여러 소스를 조합하는 단일 행동이 슬라이스 간 재사용될 때만 |
| 유틸 함수 | 순수 변환, 부수효과 없음 |

---

## 슬라이스 간 통신

| 유형 | 메커니즘 | 설명 |
| --- | --- | --- |
| 데이터 요청 | 계약(IProvider/Interface) | 응답이 필요할 때 |
| 이벤트 전파 | EventBus | 여러 슬라이스가 반응할 때 |
| 화면 이동 | Router/SceneManager | 항상 라우터 사용 |

**공통 원칙: 슬라이스는 상대방을 직접 알면 안 된다.**

## 에러 처리

Rust의 에러 처리 원칙에서 영감: **에러는 명시적이어야 하고, 호출자의 필요에 따라 상세도를 결정한다.**

핵심 질문: "호출자가 이 에러의 종류에 따라 다르게 행동하는가?"

| 구분해야 하는가? | 선택 | 예시 |
| --- | --- | --- |
| No — 로깅/표시만 | 공통 에러 | 네트워크 실패 → 메시지 표시 |
| Yes — 분기 처리 필요 | 슬라이스별 에러 | 인증 실패 → 로그아웃, 데이터 없음 → 빈 화면 |

## Cross-cutting Concerns

> 슬라이스 코드를 수정하지 않고 바깥에서 감싼다.

| 관심사 | 위치 |
| --- | --- |
| 로깅 | 플랫폼 Observer/Interceptor/Autoload |
| 네트워크 에러 | HTTP 클라이언트 Interceptor |
| 전역 에러 | 앱 진입점 (main/Application) |
| 검증 | 슬라이스 내부 + shared/utils (순수 함수만) |

## 테스트 전략

| 레벨 | 대상 | 도구 |
| --- | --- | --- |
| 단위 | Repository, ViewModel/Cubit 핵심 로직 | Fake + 단위 테스트 |
| E2E | 전체 사용자 흐름 | 플랫폼별 E2E 도구 |

**Mock 대신 Fake 사용** — Mock은 구현 세부사항에 결합됨. Fake는 실제 동작을 흉내내는 구현체.

```
test/
  fakes/          ← Fake 모음
  features/       ← 슬라이스 구조 그대로 미러링
```

## 냄새 감지

```
✘ features/가 features/를 직접 참조
✘ shared/에 구현체 또는 비즈니스 로직
✘ 글루 로직이 슬라이스 안에 있음
✘ 미래를 위해 미리 만든 피처군 shared
✘ Manager, Helper, Utils — 역할 불명확한 이름
```
