## 자명한 코드

> 근본 원칙을 코드 레벨에서 실천한다.

코드 안에 자명한 코드를 넣는 것을 선호한다.

### 명확성

- 오해의 여지를 없앤다. 예: `time`이 아니라 `timeMinutes`
- 계약 검증(사전조건, 사후조건, 불변식)으로 잘못된 값을 즉시 잡는다
- 함수 시그니처만으로 의도를 파악할 수 있어야 한다

### 가독성

- 다른 언어에서도 통하는 기본 제어문 선호 (`if/else`, `for`, 패턴 매칭)
- 언어 고유 문법이라도 단순하게 읽히면 허용 (예: 대수적 타입, null 안전 연산자)
- 제어 흐름을 숨기는 고차 함수/매크로는 신중하게 사용

### 심플함

- 예측 가능한 코드가 안전한 코드다
- 명확성을 위해 코드가 길어지는 건 허용한다. 단, 함수가 하나의 역할을 제대로 해야 한다

### 범용 실무 판단 기준

언어·플랫폼에 관계없이 적용하는 판단 기준이다.

| 상황 | 선택 | 기준 |
| --- | --- | --- |
| 고차 함수 체이닝 vs 명시적 제어문 | 명시적 제어문 | 제어 흐름을 숨기지 않는다 |
| 패턴 매칭 vs if/else if | 복잡도에 따라 | 단순 매핑이면 패턴 매칭, 로직이 있으면 if/else |
| map/filter vs for 루프 | map/filter | 불변성이 다른 로직의 침투를 막는다 |
| early return vs 단일 return | early return | 케이스를 줄여가며 읽는 게 직관적 |
| nullable vs 기본값 | 기본값 | null은 "없음"의 이유를 말해주지 않는다 |
| 원시 타입 vs 래퍼 타입 | 래퍼 타입 | 컴파일러가 타입 실수를 잡아준다 (newtype 패턴) |
| 넓은 접근 제어 vs 좁은 접근 제어 | 좁은 접근 제어 | 실수를 사전에 방지 |
| 에러: 대수적 타입 vs 예외 상속 | 대수적 타입 | 상속 지양, 컴파일러가 분기를 강제 |
| DI: 생성자 주입 vs 프로퍼티 위임 | 생성자 주입 | 의존성이 시그니처에 드러나야 명확 |
| 개별 상태 vs 단일 상태 객체 | 단일 상태 객체 | 하나의 역할에 하나의 상태 — 역할 단위 |
| 이벤트: 개별 콜백 vs 단일 이벤트 타입 | 단일 이벤트 타입 | 타입 정의 + 시그니처 안정성 |
