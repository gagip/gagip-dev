## Notion Reports DB 정보

- **Data Source ID**: 환경변수 `$NOTION_REPORTS_DS_ID`에서 읽음
- **상태 초기값**: 항상 `초안`으로 생성
- **작성일**: 오늘 날짜 (YYYY-MM-DD)

## 작업 순서

### 1. Data Source ID 결정

환경변수에서 읽는다. 값이 비어있으면 사용자에게 설정을 요청한다.

```bash
DS_ID="$NOTION_REPORTS_DS_ID"
echo $DS_ID
```

### 2. 보고서 유형 결정

argument 또는 대화 맥락에서 유형을 판단한다:
- `구현계획` — 작업 시작 전 계획 제안
- `완료보고` — 작업 완료 후 결과 보고
- `분석` — 코드/데이터/이슈 분석 결과
- `의사결정` — 선택지를 제시하고 사용자 결정 요청

### 3. 프로젝트 태그 결정 및 옵션 등록

프로젝트명은 아래 우선순위로 결정한다:

1. **사용자가 명시한 경우** — 해당 이름 그대로 사용
2. **워킹 디렉토리** — `basename $(pwd)` 또는 `git remote get-url origin 2>/dev/null`로 레포명 추출
3. **대화 맥락** — 언급된 프로젝트/서비스/앱 이름
4. **위 모두 불명확** → `기타`

`기타` 이외의 프로젝트명이 결정됐다면, DB 스키마에 해당 옵션이 있는지 확인 후 없으면 추가한다.

**옵션 존재 여부 확인:**  
`mcp__claude_ai_Notion__notion-fetch`로 Data Source ID를 조회해 `프로젝트` 컬럼의 `options` 목록에 해당 이름이 있는지 확인한다. 있으면 이 단계를 건너뛴다.

**새 프로젝트 옵션 추가 방법:**

`mcp__claude_ai_Notion__notion-update-data-source`를 호출한다:
- `data_source_id`: 1단계에서 결정한 DS_ID
- `schema_changes`:

```
ALTER COLUMN "프로젝트" ADD OPTION '<프로젝트명>':blue
```

### 4. 본문 작성

유형별 템플릿에 맞게 본문을 작성한다.

> **Notion 렌더 문법 (필수)** — Notion은 표준 마크다운의 일부를 그대로 렌더하지 않는다. 저장 전 본문을 Notion-flavored Markdown으로 변환한다:
> - **표**: 파이프(`|`) 마크다운 표는 렌더되지 않는다. 반드시 `<table>` HTML로 변환한다 (`<tr>`/`<td>`, 헤더가 있으면 `<table header-row="true">`). 셀 안엔 리치텍스트만 — `**굵게**`·`` `코드` ``는 되지만 헤딩·리스트 등 블록 요소는 안 된다.
> - **콜아웃**: 요약/상태 박스로 쓰는 `> 인용`은 `<callout icon="📌">…</callout>`로 바꾼다. 자식 블록은 탭으로 들여쓴다.
> - **구분선**: `---` 그대로 사용 가능.
> - 코드블록 밖에서 `~`·`<`·`>`·`|` 등은 `\`로 이스케이프한다 (예: `P0\~P4`). 코드블록 안은 이스케이프하지 않는다.
> - 완전한 문법은 MCP 리소스 `notion://docs/enhanced-markdown-spec`를 읽어 확인한다 — 표·콜아웃 문법을 추측하지 말 것.

#### 구현계획
`draft-plan` 스킬에서 위임된 경우 계획서 전문을 받아 **렌더 문법만 변환**해 저장한다(구조는 손대지 않는다 — 이미 리빙 문서 구조로 작성돼 있다).
직접 호출된 경우 리빙 문서 구조로 구성한다 — **계획**(목표 · 추적 매트릭스 · 결정 · 구현계획) → `---` → **진행 현황**(생성 시 빈 스캐폴드) → **변경·결정 로그**. 추적 매트릭스는 달성기준 ↔ 성공기준 ↔ 검증 ↔ 현황을 한 표로 묶는다(파이프 표는 위 규칙대로 `<table>`로 변환).
```markdown
> 📌 요약 — 한 줄 / 상태 / 문서 사용법
# 제목
## 목표
## 추적 매트릭스   (달성기준 ↔ 성공기준 ↔ 검증 ↔ 현황 — <table>로 변환)
## 결정 사항
## 구현 계획
---
## 진행 현황        (생성 시 빈 스캐폴드)
## 변경·결정 로그   (생성 결정 시드, 최신 위로)
```

#### 완료보고
```
> 요약: (한 줄 요약)
> 요청: [확인]

## 작업 내용 요약
## 결과물
## 이슈 / 특이사항
## 다음 단계 제안
```

#### 분석
```
> 요약: (한 줄 요약)
> 요청: [검토] / [방향 조언]

## 분석 배경
## 분석 결과
## 인사이트
## 제안
```

#### 의사결정
```
> 요약: (한 줄 요약)
> 요청: [선택]

## 결정이 필요한 상황
## 선택지
  - 각 선택지별 장단점
## 추천안 + 근거
## 결정 기한 (있다면)
```

### 5. Notion 페이지 생성

`mcp__claude_ai_Notion__notion-create-pages`를 호출한다.

- `parent.type`: `data_source_id`
- `parent.data_source_id`: 1단계에서 결정한 DS_ID
- `properties`:
  - `제목`: 보고서 제목
  - `유형`: 결정된 유형
  - `상태`: `초안`
  - `요약`: 한 줄 요약
  - `프로젝트`: 3단계에서 결정한 프로젝트명 (JSON 배열, 예: `["my-app"]`)
  - `date:작성일:start`: 오늘 날짜 (YYYY-MM-DD)
- `content`: 위에서 작성한 본문

### 6. 완료 메시지 출력

```
📋 보고서가 Notion에 작성됐습니다.

- 제목: <보고서 제목>
- 유형: <유형>
- 상태: 초안
- 프로젝트: <프로젝트명>
- URL: <생성된 페이지 URL>

Notion에서 검토 후 상태를 변경해주세요.
```

## 행동 원칙

- 보고서는 항상 `초안` 상태로 생성한다. 승인은 사용자가 직접 한다.
- 본문은 간결하게 — 핵심만 담고 불필요한 내용은 넣지 않는다.
- 사용자가 프로젝트명을 언급했다면 반드시 그 이름을 사용한다.
- **코드를 다루는 보고서(분석·완료보고·구현계획)는 줄번호 나열 금지.** `경로:라인`만 주면 독자가 IDE로 찾아야 해 번거롭다. 현황·문제를 짚을 땐 핵심 코드를 발췌해 코드블록으로 넣고 말로 설명한다. 파일명은 발췌 출처로만 붙인다. 보고서만 읽어도 ① 배경(도메인·용어·흐름) ② 문제(지금 코드가 무엇을 어떻게 해서 왜 문제인지, 발췌로 증명) ③ 하고자 하는 것을 파악할 수 있게 자립적으로 구성한다. (`draft-plan`의 "보고서 작성 원칙"과 동일 기조)
- **표·콜아웃은 렌더 문법으로 변환해 저장한다.** 파이프 마크다운 표와 `> 콜아웃`은 Notion에서 깨진다 — `<table>`/`<callout>`로 바꾼다 (§4 Notion 렌더 문법).
- **기존 보고서의 진행 현황·로그를 갱신**하는 흐름은 이 스킬(생성 전용) 범위 밖이다 — `notion-update-page`로 별도 처리한다.
