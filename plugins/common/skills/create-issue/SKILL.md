## 입력

```
$ARGUMENTS
```

입력이 없으면 사용자에게 이슈 내용(제목, 타입, 설명)을 물어볼 것.

---

## 작업 순서

### 1. 이슈 내용 파악

- `$ARGUMENTS`로 충분한 정보가 있으면 바로 진행
- 부족하면 사용자에게 질문 (한 번에 필요한 것 모두 물어볼 것)

### 2. 레포 라벨 감지

```bash
gh label list --limit 100 --json name --jq '.[].name'
```

- 위 명령으로 현재 레포에 실제 존재하는 라벨 목록을 확인
- 이후 단계에서 이 목록에 있는 라벨만 사용할 것 (없는 라벨은 붙이지 않음)

### 3. 타입 및 라벨 결정

아래 매핑을 참고해 타입과 라벨을 결정하되, **2단계에서 확인한 실제 라벨 목록에 존재하는 것만 사용**:

| 타입          | 권장 라벨         |
|-------------|-----------------|
| `feat:`     | `enhancement`   |
| `fix:`      | `bug`           |
| `refactor:` | `refactor`      |
| `style:`    | `style`         |
| `docs:`     | `documentation` |
| `test:`     | `test`          |
| `chore:`    | `chore`         |
| `ai:`       | `ai`            |

- 필요시 추가 라벨: `priority:high`, `priority:low` (존재하는 경우에만)

### 4. 이슈 본문 구성

```markdown
## 개요
## 구현 범위
## 기술 구현
## 작업 목록
- [ ] 작업 1
## 참고
```

### 5. 사용자 피드백 후 대기 [STOP]

- 구성한 이슈 본문을 사용자에게 출력하고 **반드시 여기서 멈출 것**
- 수정 요청이 오면 본문을 수정한 뒤 다시 대기할 것
- 사용자가 명시적으로 생성을 지시("올려줘", "생성해줘", "만들어줘" 등)하기 전까지 6단계를 절대 실행하지 말 것
- 스킬 호출 자체를 이슈 생성 승인으로 간주하지 말 것

### 6. 이슈 생성 [사용자 명시적 승인 후에만 실행]

```bash
gh issue create --title "<타입>: <제목>" --label "<라벨>" --body "$(cat <<'EOF'
<이슈 본문>
EOF
)"
```

- 생성된 이슈 URL 안내

## 행동 원칙

- 스킬 호출 자체를 이슈 생성 승인으로 간주하지 않는다 — 반드시 사용자 명시적 지시 후에만 생성한다
- 존재하지 않는 라벨은 절대 사용하지 않는다 — 항상 실제 레포 라벨 목록에서 선택한다
