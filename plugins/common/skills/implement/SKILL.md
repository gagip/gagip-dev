## 입력

```
$ARGUMENTS
```

## 규칙

- 구현 전 반드시 계획을 먼저 출력하고 승인 대기
- 파일 수정 전 반드시 해당 파일 Read
- 한 번에 모든 파일을 수정하지 말고, 계획에서 정한 순서대로 진행
- 커밋은 사용자가 명시적으로 요청할 때만 진행

---

## 작업 순서

### 1. 러버덕 설계 세션 [STOP × N]

AI가 먼저 판단하지 않는다. 사용자가 먼저 생각을 꺼내도록 유도하고, 질문으로 설계를 함께 완성한다.

#### 1-1. 첫 질문으로 시작

`$ARGUMENTS`가 있으면 그 맥락을 짧게 확인한 뒤, 아래 형식으로 대화를 연다:

```
[기능명]을 구현하려고 하시는군요.

어떻게 접근하실 생각인가요? 지금 떠오르는 방향이 있다면 자유롭게 말씀해 주세요.
(완성된 아이디어가 아니어도 됩니다)
```

#### 1-2. 소크라테스식 질문으로 설계 탐구

사용자 답변을 듣고 아래 방식으로 대화를 이어간다:

- **왜 질문**: "왜 그 방식을 선택하셨나요?" — 전제를 드러냄
- **엣지 케이스 질문**: "X 상황에서는 어떻게 되나요?" — 구멍을 발견
- **관계 질문**: "기존 Y 컴포넌트와는 어떻게 연결되나요?" — 의존성 파악
- **대안 질문**: "Z 방식도 가능한데, 어떻게 생각하세요?" — 막혔을 때만

규칙:
- 한 번에 질문 하나만
- AI 의견은 사용자가 막히거나 명시적으로 요청할 때만 제시
- 관련 코드가 필요하면 Glob·Grep으로 확인한 뒤 발췌해서 보여줌 (`파일경로:라인번호` 형식으로 위치 명시)

#### 1-3. 설계 수렴 확인

대화를 통해 핵심 결정이 충분히 논의됐다고 판단되면:

```
지금까지 나온 설계를 정리해볼까요?

- [결정 1]
- [결정 2]
...

계획으로 넘어가도 괜찮을까요?
```

사용자가 확인하면 2단계로 진행.

---

### 2. 구현 계획 수립 후 승인 요청 [STOP]

1단계 대화 결과를 반영하여 계획 출력 후 사용자 승인 대기:

```
## 구현 계획: <기능명>

### 영향 범위
| 파일 | 작업 |
|------|------|
| `경로/파일명` | 생성 / 수정 / 삭제 |

### 핵심 설계 결정
- (1단계 대화에서 합의된 결정 반영)

### 구현 순서
1. <첫 번째 작업>
2. <두 번째 작업>
...

진행할까요?
```

- 승인 → 3단계 진행
- 수정 요청 → 계획 수정 후 다시 대기

### 3. 구현

- 계획에서 정한 순서대로 파일 수정·생성
- 수정 전 반드시 해당 파일 Read
- 코드 언급 시 항상 `파일경로:라인번호` 형식으로 위치 명시

### 4. 코드 리뷰 및 피드백 [STOP]

- 구현한 내용을 요약하고 주요 결정 사항을 사용자에게 출력한 뒤 **반드시 여기서 멈출 것**
- 사용자 추가 피드백 수렴 후 수정 여부 결정
- 커밋이 필요하면 사용자에게 `commit` 스킬 사용을 안내
