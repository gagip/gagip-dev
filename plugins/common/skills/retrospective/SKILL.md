# 회고 스킬

현재 대화 세션을 돌아보며 유의미한 정보를 추출하고, 적절한 위치에 기록을 남긴다.
다음 대화에서 LLM이 같은 실수를 반복하지 않거나, 더 나은 판단을 내릴 수 있도록 맥락을 축적하는 것이 핵심 목적이다.

---

## 실행 절차

### Step 1: 대화 세션 분석

현재 대화 전체를 읽으며 아래 항목을 추출한다.
여러 세션을 대상으로 하는 회고라면, 먼저 아래 "옵션: 범위 확장 → 다중 세션·주간 회고 워크플로우"로
세션을 분산 분석한 뒤 그 결과를 종합해 같은 항목을 추출한다.

**추출 항목**:
- **수정/재작업 발생 지점**: 내가 처음 제안한 것을 사용자가 수정 요청한 경우 → 왜 틀렸는지
- **사용자 명시적 피드백**: "그건 아니야", "그게 아니라", "다시 해봐", "원본 유지해" 등의 표현
- **사용자 암묵적 피드백**: 수정 없이 바로 수락한 접근법 → 무엇이 잘 작동했는지
- **새로 파악한 사실**: 프로젝트 구조, 팀 맥락, 사용자 선호 등 이전에 몰랐던 것
- **스킬/문서 개선 포인트**: 스킬이 놓친 케이스, 문서에 없는 규칙
- **반복 작업 패턴**: 이번 세션(또는 여러 세션)에서 여러 번 반복한 다단계 절차 → 신규 스킬 후보
- **문서화 공백**: 내린 결정·파악한 구조·확립한 절차인데 코드·CLAUDE.md·볼트·문서 어디에도 안 적힌 것 → 문서화 후보

### Step 2: 인사이트 분류 및 업데이트 항목 결정

추출한 항목을 아래 4가지 카테고리로 분류하고, 각각 업데이트가 필요한지 판단한다.
**업데이트 여부는 강제하지 않는다** — 유의미한 변경이 없으면 해당 항목은 건너뛴다.

| 카테고리 | 무엇을 추출하나 | 어디로 보내나 |
|---|---|---|
| 스킬/지침 개선 | 스킬이 놓친 케이스, 잘못된 절차 | 해당 `SKILL.md` 직접 수정 |
| 신규 스킬 추천 | 여러 번 반복한 다단계 작업 패턴 | 사용자에게 스킬화 제안 (동의 시 `skill-creator`로 연결) |
| 메모리 | 사용자 선호·피드백·프로젝트 맥락 | **적격성 검증 게이트 통과분을 자동으로 `memory-curator`에 전달** — 저장·중복판정은 memory-curator가 담당 |
| 볼트 (회사) | 업무 회고 — 의사결정·문제 해결·팀 맥락 | 회사 볼트 `retrospective/YYYY-MM-DD.md` |
| 볼트 (개인) | 회사 맥락 뺀 순수 기술 인사이트 | 개인 볼트 `retrospective/YYYY-MM-DD.md` |
| 문서화 공백 | 결정·구조·절차인데 어디에도 미기록 | 사용자에게 문서화 제안 (볼트 wiki·프로젝트 docs·README 등) |
| CLAUDE.md 지침 | 지속 적용할 규칙 | 계층 판단 후 적절한 `CLAUDE.md` / `CLAUDE.local.md` (아래 기준) |

### 업데이트 판단 기준

카테고리별로 아래 기준을 충족할 때만 처리한다.

**메모리 → 적격성 검증 게이트 통과분을 자동으로 `memory-curator`에 전달**
- retrospective는 메모리 "후보"(사용자가 교정한 접근법, 비자명한 선호, 기존에 없는 새 패턴)를 식별한 뒤, 아래 **적격성 검증 게이트**로 거른다. 메모리는 다른 카테고리와 달리 **사용자 확인 없이 자동 전달**한다(Step 4 참조) — 대신 이 게이트가 품질을 보장한다.
- **적격성 검증 게이트** — 후보가 아래를 **모두** 충족해야 전달한다. 하나라도 걸리면 제외하고, 제외 사유를 회고 보고에 남긴다.
  1. **재사용성**: 세션을 넘겨 다음 판단·행동을 바꾼다 (이 대화 한정·일회성이면 제외)
  2. **비자명성**: 코드·git·CLAUDE.md·기존 메모리(MEMORY.md)에 이미 있지 않다
  3. **durable**: 추측·임시 상태가 아니라 확정된 사실/선호/맥락이다
  4. **타입 적합**: user/feedback/project/reference 중 하나에 명확히 부합한다
  5. **행동 변화 한 줄**: "다음에 이걸로 무엇을 다르게 할지"가 한 줄로 나온다
- 게이트 통과분은 `memory-curator` 기본 모드를 **자동 실행**해 넘긴다(인자로 후보 목록·타입·Why/How를 전달). 실제 저장·2차 중복판정·MEMORY.md 인덱스 관리는 memory-curator가 담당한다.
- 왜 이중 게이트인가: retrospective 게이트는 "전달할 가치가 있는가"(품질)를, memory-curator 필터(이미 저장됨 / 코드에 있음 / CLAUDE.md에 있음)는 "중복인가"(저장 관문)를 본다. 역할이 달라 겹치지 않는다. retrospective는 여전히 **직접 저장하지 않으며**, memory-curator가 메모리의 단일 저장 관문으로 남는다.

**신규 스킬 추천** — 같은 다단계 작업을 반복했을 때
- 한 세션에서 같은 절차를 여러 번 반복했거나, 기간 회고에서 여러 세션에 걸쳐 같은 순서로 도구를 엮는 패턴이 보일 때.
- **추천만 한다(자동 생성 금지)**. "이런 패턴이 반복되니 스킬로 만들면 좋겠다"고 제안하고, 사용자가 동의하면 `skill-creator`로 넘긴다.
- 왜: 반복은 스킬화의 가장 강한 신호고, 회고는 그 신호를 포착하기 가장 좋은 시점이다.

**문서화 공백** — 결정·구조·절차가 어디에도 안 적혔을 때
- 이번 작업에서 내린 의사결정·파악한 시스템 구조·확립한 절차인데, 코드 주석·CLAUDE.md·볼트·README·이슈 어디에도 기록이 없을 때.
- **추천만 한다.** 귀착지를 함께 제안한다 — 재사용 지식이면 볼트 wiki, 프로젝트 규칙이면 프로젝트 `docs`/`README`/`CLAUDE.md`, 일회성 작업 보고면 `private/`.
- 왜: 머릿속에만 있는 결정은 다음 사람(미래의 나 포함)에게 전달되지 않는다.

**볼트 (회사)** — 업무 관련 기록, 나중에 사람이 다시 꺼내볼 가치가 있을 때
- 의사결정 근거 (왜 이 방식을 선택했는지, 팀/프로젝트 맥락 포함)
- 오늘 작업에서 발생한 문제와 해결 과정
- 반복 사용 가능한 업무 체크리스트나 절차

**볼트 (개인)** — 회사 맥락을 제거한 순수 기술 인사이트
- 범용 기술 지식 (예: iOS 코드사이닝 절차, 머지된 PR 이후 누락 커밋 복구)
- 특정 회사/팀/프로젝트를 언급하지 않아도 성립하는 내용만

**스킬/지침** — 스킬이 다음에 같은 상황에서 다르게 행동해야 할 때
- 스킬이 커버하지 못한 케이스가 실제로 발생한 경우
- 절차상 순서가 잘못됐거나 빠진 단계가 있는 경우

**CLAUDE.md — 계층을 구분해서 반영한다**
지침이 적용될 범위에 맞는 층을 고른다. 잘못된 층에 넣으면 너무 넓게(전역) 새거나 너무 좁게(디렉토리) 갇힌다.

| 층 | 파일 | 적용 범위 | 공유 | 언제 |
|---|---|---|---|---|
| 전역 | `~/.claude/CLAUDE.md` | 모든 프로젝트·대화 | 개인 환경 | 보편 규칙. "항상 이렇게 해줘" |
| 프로젝트 | `<repo>/CLAUDE.md` | 그 레포 전체 | 팀 공유(git) | 그 프로젝트의 팀 규칙·구조 |
| 디렉토리 | `<repo>/<subdir>/CLAUDE.md` | 그 하위 트리 | 팀 공유(git) | 모노레포 하위 패키지 등 국소 규칙 |
| 개인 | `<repo>/CLAUDE.local.md` | 그 레포, 나만 | git 미추적 | **개인 지침** — 비공유 작업 방식·로컬 경로·개인 선호 |

- **판단 기준 = 공유 여부**: 팀과 공유할 규칙이면 `CLAUDE.md`(전역/프로젝트/디렉토리), 다른 팀원에게 강요하면 안 되는 **나 개인만의 방식·로컬 환경**이면 `CLAUDE.local.md`.
- 전역 변경은 파급이 가장 크니 가장 신중하게 — 정말 모든 프로젝트에 맞을 때만. 사용자가 명시적으로 "항상 이렇게"라고 한 경우.

### Step 3: 회고 요약 출력

업데이트 전, 아래 형식으로 회고 내용을 먼저 출력한다.

```
## 회고 요약 — YYYY-MM-DD

### 잘 된 것
- ...

### 개선이 필요했던 것
- 무엇이 문제였는지 + 왜 그랬는지

### 추천 업데이트
- [ ] [스킬] daily-standup: "다듬기 모드" 분기 추가
- [ ] [신규 스킬?] 같은 다단계 작업을 N회 반복 → 스킬화 제안 (skill-creator)
- [x] [메모리·자동저장] 사용자 원본 존중 선호 (적격성 게이트 통과 → memory-curator 자동 전달)
- [ ] [메모리·제외] 이번 PR 구현 세부 (사유: 코드에 있음 — 게이트 2 불충족)
- [ ] [볼트] 오늘 배운 GitHub 이슈 탐색 패턴 기록
- [ ] [문서화?] X 절차가 어디에도 미기록 → 볼트 wiki 제안
- [ ] [CLAUDE.local.md] 개인 로컬 경로/작업 방식 선호
- [ ] [CLAUDE.md] (변경 없음)
```

각 항목마다 **왜** 업데이트가 필요한지 한 줄로 근거를 제시한다.
업데이트 불필요 항목은 "(변경 없음)"으로 명시한다.

### Step 4: 업데이트 실행 (메모리는 자동, 나머지는 사용자 확인)

**메모리**는 Step 2의 적격성 검증 게이트 통과분을 **사용자 확인 없이 자동으로** `memory-curator`에 전달해 저장한다. (게이트가 품질을, memory-curator가 중복을 거른다.) 게이트 통과/제외 결과는 회고 요약·완료 보고에 투명하게 남긴다.

**메모리를 제외한 나머지**(스킬·볼트·문서화·CLAUDE.md·신규 스킬)는 추천 목록을 보여주고 사용자 확인을 받은 뒤 실행한다.

> "메모리는 적격성 검증 후 자동 저장했습니다. 나머지 항목을 업데이트할까요? 빼거나 추가할 항목이 있으면 말씀해 주세요."

사용자가 전체 동의하거나 일부 선택하면 해당 항목만 실행한다.

**업데이트 원칙**:
- **스킬 문서**: 기존 문서를 최대한 유지하고, 변경 최소화. 새 규칙은 기존 구조에 자연스럽게 삽입
- **메모리**: 직접 저장하지 않는다. 적격성 검증 게이트 통과분을 `memory-curator` 기본 모드로 **자동 전달**해 저장·중복판정·MEMORY.md 인덱스 일치 검증을 일임한다(사용자 확인 불요). 게이트에서 제외한 후보는 보고에 사유와 함께 남긴다. (중복 정리·전역 승격 후 메모리 삭제 같은 위생 작업도 memory-curator가 담당.)
- **신규 스킬 추천 / 문서화 공백**: 제안만 하고 사용자 동의를 받는다. 스킬화는 `skill-creator`로 연결, 문서화는 함께 제안한 귀착지에서 진행.
- **볼트 (회사)**: CLAUDE.md에 명시된 회사 볼트의 `retrospective/YYYY-MM-DD.md`에 저장
- **볼트 (개인)**: CLAUDE.md에 명시된 개인 볼트의 `retrospective/YYYY-MM-DD.md`에 저장 (회사 맥락 제거 후)
- **CLAUDE.md / CLAUDE.local.md**: 위 계층 표대로 올바른 층을 고른다. 전역 변경은 사용자가 명시적으로 동의한 경우만.

### Step 5: 완료 보고

업데이트한 파일 목록과 변경 내용을 간략히 보고한다.

## 옵션: 범위 확장

기본은 현재 대화 세션만 분석한다. 사용자가 원하면 아래 추가 소스도 포함할 수 있다.

| 옵션 | 활성화 방법 | 추가 작업 |
|---|---|---|
| 오늘 git log | "git 커밋도 포함해줘" | 워크스페이스 하위 레포의 오늘 커밋 수집 |
| 오늘 전체 세션 | "오늘 전체 세션 다 봐줘" | 오늘자 세션 JSONL 모아 분석 (아래 워크플로우) |
| 주간/기간 회고 | "지난 주 회고", "최근 N일" | 기간 내 여러 프로젝트 세션 전체를 분산 분석 (아래 워크플로우) |

### 다중 세션·주간 회고 워크플로우

"지난 주", "이번 주", "최근 N일", "여러 세션" 같은 요청은 세션이 수십~수백 개라
현재 컨텍스트로 한 번에 읽을 수 없다. 아래 순서로 분산 처리한다.

**1. 세션 수집** — 대상 기간의 실질 세션을 모은다. 세션 파일은
`~/.claude/projects/<인코딩된-경로>/*.jsonl` 에 있다 (`subagents/` 하위는 제외).
`daily-summary` 스킬에 `collect_jsonl.py` 가 있으면 쓰고, 없으면 직접 수집한다:

```bash
# 기간 내 실질 세션을 날짜|줄수|경로로. mtime이 아니라 첫 timestamp로 날짜를 잡는다.
for f in $(find ~/.claude/projects -path "*subagents*" -prune -o -name "*.jsonl" -mtime -16 -print | grep -i <워크스페이스>); do
  ts=$(grep -m1 -o '"timestamp":"[0-9-]*' "$f" | head -1 | sed 's/"timestamp":"//')
  echo "$ts|$(wc -l < "$f")|$f"
done | awk -F'|' '$1 >= "<시작일>" && $2 > 20' | sort
```
> 20줄 이하는 즉시 종료·취소된 빈 세션이니 제외한다. zsh에서 문자열 비교는 `[[ ]]` 또는 `awk`로 한다.

**2. 날짜별 병렬 분산 분석** — 날짜(또는 프로젝트) 단위로 묶어 서브에이전트(`Explore` 등)에 분배한다.
각 에이전트는 맡은 세션들에서 **사용자 발화**만 뽑아 회고 재료를 추출한다 (tool_result는 자동 제외됨):

```bash
jq -r 'select(.type=="user") | .message.content | if type=="string" then . elif type=="array" then (.[]|select(.type=="text")|.text) else empty end' <세션.jsonl>
```
> jsonl 전체를 읽지 말고 위 jq로 사용자 발화에 집중한다. assistant 출력은 결론만 일부 참고한다.
> 각 에이전트에 Step 1의 추출 항목과 "날짜별 마크다운" 출력 형식을 그대로 지시한다.

**3. 실제 산출물 대조** — 각 레포 `git log --since=<시작일>` 으로 커밋/PR을 모아 교차 검증한다.
세션은 "왜/어떻게"(과정·피드백), git log는 "무엇을"(성과)을 보여준다.

**4. 종합** — 에이전트 결과를 합쳐 **반복되는 선호 패턴**을 추린다.
주간 회고에서는 개별 사건보다 반복 패턴이 가장 큰 자산이다.

## 볼트 저장 형식

볼트 경로는 CLAUDE.md에 명시된 값을 사용한다.

### 회사 볼트 — `retrospective/YYYY-MM-DD.md`

```markdown
# 회고 — YYYY-MM-DD

## 잘 된 것
- ...

## 개선이 필요했던 것
- 문제 상황: ...
- 원인: ...
- 다음엔: ...

## 업데이트한 것
- [스킬] daily-standup — 다듬기 모드 분기 추가
- [메모리] feedback-original-content-respect.md 신규 생성
```

### 개인 볼트 — `retrospective/YYYY-MM-DD.md`

회사/팀/프로젝트명 없이 범용 기술 지식만 작성한다.

```markdown
# 기술 인사이트 — YYYY-MM-DD

## 오늘 배운 것
- ...

## 재사용 가능한 절차/체크리스트
- ...
```

## 주의사항

1. **강제 업데이트 금지**: 유의미한 변경이 없으면 파일을 건드리지 않는다. "변경 없음"이 정직한 결과다.
2. **과잉 해석 금지**: 사용자가 단순히 방향을 바꾼 것과 내가 틀린 것을 구별한다. 사용자 의도 변경은 피드백이 아니다.
3. **볼트 보안 원칙 준수**: 회사 내부 기밀(코드, 설계, API 키 등)은 반드시 회사 볼트(`synergyai-obsidian`)에만 저장한다.
4. **메모리는 적격성 게이트 후 자동으로 memory-curator에 전달**: retrospective는 메모리를 직접 저장하지 않는다. 적격성 검증 게이트(재사용성·비자명성·durable·타입적합·행동변화)를 통과한 후보만 **사용자 확인 없이 자동** 전달하고, 중복 방지·인덱스 검증·전역 승격 후 정리는 memory-curator의 책임이다. retrospective 게이트(품질)와 memory-curator 필터(중복)가 역할을 나눠 같은 세션을 두 번 긷지 않게 한다.
5. **추천은 추천으로 끝낸다**: 신규 스킬·문서화는 제안만 하고, 사용자 동의 없이 스킬을 만들거나 문서를 쓰지 않는다.
6. **CLAUDE.md 계층 신중하게**: 층 선택을 틀리면 너무 넓게/좁게 퍼진다. 특히 전역 변경은 파급이 크니 사용자가 명시적으로 원하는 경우에만. 개인 전용은 `CLAUDE.local.md`로.
