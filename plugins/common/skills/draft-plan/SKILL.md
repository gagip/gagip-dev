# Draft Plan 스킬

GitHub 이슈 또는 자연어 주제를 기반으로 논의·QA를 진행하며 구현 계획서를 작성하고,
`notion-report` 스킬을 통해 Notion Reports DB(`$NOTION_REPORTS_DS_ID`)에 보고서로 저장하는 스킬이다.

**핵심 원칙**:
- 이 스킬은 **문서 편집 도구**다. 코드를 수정하거나 구현을 시작하지 않는다.
- Plan Mode는 계획서 내용을 작성·수정할 때마다 자유롭게 진입·종료할 수 있다.
- Plan Mode 승인 = `notion-report` 스킬로 Notion 저장. 구현 시작이 아니다.
- 구현은 논의가 끝난 뒤 `/clear`로 컨텍스트를 초기화하고 새 세션에서 시작한다.

---

## 실행 절차

### Step 1: 입력 파싱

인자 형태에 따라 다르게 처리한다.

**이슈 번호** (`#35`, `35`, `gh-35`):
```bash
gh issue view <number> --json number,title,body,labels,url
```
가져온 `number`, `title`, `body`, `labels`, `url` 을 이후 분석 컨텍스트로 보유한다.

**GitHub URL** (`https://github.com/.../issues/35`):
owner/repo/number를 파싱해 위와 동일하게 `gh issue view` 로 조회한다.

**자연어 주제** (예: "로그인 화면 개선"):
그대로 주제 문자열로 사용하고 이슈 메타데이터는 없는 것으로 처리한다.

**인자 없음**:
사용자에게 무엇에 대한 계획서인지 묻는다.

**기존 계획서 파일 경로** (`@path/to/plan.md` 등 로컬 계획서 .md):
파일을 Read로 읽은 뒤 아래 항목을 점검하고, 누락된 항목을 사용자에게 알린 다음 보강 여부를 확인한다.
- YAML 프론트매터 (`created`, `status`) 존재 여부
- Context 섹션에 브랜치명 명시 여부
- `## 커밋 계획` 섹션 존재 여부
- `## 검증` 섹션의 자동/수동 체크 분리 여부
- `## 코드 리뷰` 섹션 존재 여부 (`/code-review medium` + 위험 지적만 수정 + 커스텀 리뷰 조건부/명시)
보강이 필요하면 Step 2 논의로 진입하고, 이미 완비되어 있으면 그대로 논의를 이어간다.

**git 레포가 아닌 디렉터리**:
브랜치명 안내를 생략하고 계속 진행한다. 저장은 Notion Reports DB이므로 로컬 경로와 무관하다.

---

### Step 2: 논의 및 Plan Mode 반복 진입

이 스킬은 **논의·QA 중심**으로 동작한다. Plan Mode는 계획서 내용을 정리·수정할 때만 진입하고, 논의는 일반 대화로 진행한다.

**논의 단계 (일반 대화)**:
- 사용자 질문에 답하고, 방향을 함께 결정한다.
- 코드베이스 파일·심볼·패턴을 Read/grep으로 파악해 근거를 제시한다.
- 이슈 본문에서 언급한 파일·심볼이 실제 코드에 없으면 즉시 피드백한다.
- 불명확한 부분은 `AskUserQuestion`으로 확인한다. 임의 결정하지 않는다.
- 구현 방식에 갈림길이 있으면 한 가지로 단정하지 말고, 선택을 가르는 기준을 세운 뒤 그에 따른 방안을 장단점과 함께 제시해 사용자가 고르게 한다 (추천안을 첫 번째로). 방안 개수는 고정하지 않는다 — 실질적 대안이 없으면 "대안 없음"을 밝힌다.

**Plan Mode 진입 시점**:
- 논의 중 계획서를 새로 작성하거나 수정할 내용이 생겼을 때 `EnterPlanMode`로 진입한다.
- 승인(`ExitPlanMode`) 후 다시 논의가 필요하면 일반 대화로 돌아간다.
- Plan Mode는 필요한 만큼 여러 번 반복할 수 있다.

**이슈가 있는 경우**: 이슈 본문의 "문제 / 작업 범위 / 완료 기준" 섹션을 계획서 골격으로 활용한다.

### Step 3: 계획서 본문 작성

Plan Mode 임시 플랜 파일(`~/.claude/plans/<slug>.md`)에 아래 골격으로 작성한다.
골격은 가이드일 뿐이므로 상황에 맞게 섹션을 추가·생략한다.

```markdown
# <한 줄 제목>

## 한눈에
- 이 작업이 무엇인지 1~2문장 — 배경지식 없는 독자도 핵심을 잡게

## 배경지식
- 도메인·용어·데이터 흐름을 모르는 사람용으로 풀어서 (용어는 처음 쓸 때 한 줄로 풀이)

## Context
- 이슈: #<n> / <url> (있으면)
- 변경의 이유 / 배경
- 기대 결과

## 현황 분석 (지금 무엇이 문제인가)
- 문제를 **코드 발췌**로 증명한다 — 줄번호(`경로:라인`)만 주지 말고, 관련 코드를 발췌해 코드블록으로 넣고 말로 설명한다. 파일명은 발췌 출처로만 붙인다
- 재사용 후보 (기존 함수·유틸) — 가능하면 핵심 코드를 발췌
- 현재 구조의 문제점

## 결정 사항 / 미해결 질문
- 사용자와 합의된 선택
- **대안 비교**: 구현 방식에 갈림길이 있으면 방안을 표(방안 / 장점 / 단점 / 영향)로 비교하고, 선택한 안과 근거를 남긴다
- TBD 항목 (명시적으로 남김)

## 구현 계획
1. ...
2. ...

## 검증

자동 체크 (스크립트·CI로 실행 가능):
- [ ] 항목 (명령어 또는 테스트 파일 명시)

수동 체크 (실기기·시뮬레이터·눈으로 확인):
- [ ] 항목 (자동화 불가한 것만)

> 자동화 가능한 것은 최대한 자동 체크로 분류한다.
> 수동 체크는 실기기·브라우저·시각 확인이 반드시 필요한 경우에만 남긴다.

## 영향 범위
- 변경 파일 목록
- 잠재 리스크

## 커밋 계획
- 커밋 단위별로 나열 (각 커밋은 컴파일 가능 상태 유지)
- 실제 커밋 메시지 작성은 `/syai-commit` 스킬 사용

## 코드 리뷰
구현·검증·커밋을 모두 마친 뒤, 최종 게이트로 코드 리뷰를 수행한다.
- [ ] `/code-review medium` 실행
- [ ] 프로젝트에 커스텀 코드 리뷰 스킬이 있으면 함께 실행 (탐지되면 스킬명 명시)
- [ ] 지적사항 중 **위험한 것만** 수정 — 정확성 버그·보안·데이터 손상·크래시 유발 등. 단순 정리·스타일·효율 제안은 반영하지 않거나 별도 이슈로 분리한다
```

**보고서 작성 원칙 — 코드 발췌·자립 구성**

보고서는 **코드를 직접 열어보지 않는 독자(본인·리뷰어)도 단독으로 이해**할 수 있어야 한다. 사용자는 보고서만 읽고 코드를 열어보지 않는다.

1. **줄번호 대신 코드 발췌** — `경로:라인` 줄번호만 주면 독자가 IDE를 켜서 찾아야 해 번거롭다. 현황·문제를 짚을 땐 해당 코드를 **발췌해 코드블록으로 넣고** 말로 설명한다. 파일명은 발췌가 어디서 왔는지 맥락으로만 붙인다.
2. **자립적 구성** — 보고서만 읽어도 ① 배경지식(도메인·용어·데이터 흐름) ② 문제(지금 코드가 무엇을 어떻게 하고 있어 왜 문제인지, 발췌로 증명) ③ 구현하고자 하는 것(목표·해결 방향)을 파악할 수 있게 한다. 맨 앞 "한눈에 / 배경지식" 섹션에서 용어와 흐름을 먼저 깐다.
3. 재사용할 기존 패턴도 가능하면 핵심 코드를 발췌해 "이미 이런 게 있다"를 눈으로 보이게 한다.

**코드 리뷰 섹션 작성 — 커스텀 스킬 탐지 (하이브리드)**

`## 코드 리뷰`는 구현 세션이 모든 작업·커밋을 끝낸 뒤 실행할 마지막 단계다.
draft-plan은 리뷰를 직접 실행하지 않고, 아래 항목을 계획서에 적기만 한다.

1. **빌트인 — 항상 `medium` 으로 명시**
   ```markdown
   - [ ] `/code-review medium` 실행
   ```
   `medium` 은 적은 수의 고신뢰 지적만 내므로 노이즈가 적다. 변경 규모가 매우 크면
   구현 세션 판단으로 `/code-review high` 까지 올려도 된다. PR 코멘트가 필요하면 `--comment` 를 붙인다.

2. **커스텀 — "있다면" 조건부**
   현재 세션에서 사용 가능한 스킬 목록을 살펴, 빌트인 `/code-review` 외에
   프로젝트·팀 전용 코드 리뷰 스킬(예: `review-code`)이 있는지 확인한다.
   불확실하면 `.claude/skills/` 와 설치된 플러그인을 grep으로 추가 확인한다.
   - **탐지 성공** → 스킬 이름을 그대로 명시한다.
     ```markdown
     - [ ] `/review-code` 실행 후 함께 검토
     ```
   - **탐지 실패/불확실** → 조건부 문구를 남겨 구현 세션이 판단하게 한다.
     ```markdown
     - [ ] 프로젝트에 커스텀 코드 리뷰 스킬이 있으면 함께 실행
     ```

3. **수정 범위 — 위험한 지적만**
   ```markdown
   - [ ] 지적사항 중 위험한 것만 수정 — 정확성 버그·보안·데이터 손상·크래시 유발 등. 단순 정리·스타일·효율 제안은 반영하지 않거나 별도 이슈로 분리한다
   ```
   `--fix` 는 모든 지적을 무차별로 working tree 에 적용하므로 쓰지 않는다. 위험한 지적만
   골라 손으로 반영해야 의도치 않은 리팩터링 churn 을 막는다.

> 코드 리뷰는 "모든 작업을 마친 뒤"의 게이트다. 계획서는 실행 지시만 담고,
> 실제 리뷰·수정은 `/clear` 이후 구현 세션이 수행한다.

### Step 4: 브랜치명 결정

계획서 본문 작성 중 또는 직후, 이 작업을 위한 브랜치명을 결정하여 계획서에 포함한다.

**브랜치명 생성 규칙** (`git-convention` 스킬 컨벤션 준수):

| 상황 | 형식 | 예시 |
|------|------|------|
| 이슈 있음 + 레이블 `fix`/`bug` | `fix/<number>-<title-slug>` | `fix/35-measure-screen-back-press` |
| 이슈 있음 + 그 외 레이블 | `feat/<number>-<title-slug>` | `feat/36-ecg-offline-queue` |
| 자연어 주제 | `feat/<topic-slug>` | `feat/login-screen-improvement` |

- slug: 영문 소문자 + 하이픈, 40자 이하, 한글은 영문 키워드 추출
- 이슈 레이블에 `fix`, `bug`, `hotfix` 가 있으면 `fix/` 접두사 사용
- 프로젝트에 Jira 이슈 키(`DEV-123`)가 있다면 `feat/DEV-123-<slug>` 형식 우선

결정된 브랜치명과 커밋 컨벤션을 계획서 Context 섹션에 다음과 같이 명시한다:
```markdown
## Context
- 이슈: #35 / <url>
- 브랜치: `fix/35-measure-screen-back-press`
  ```bash
  git checkout -b fix/35-measure-screen-back-press
  ```
- 변경의 이유 / 배경
```

---

### Step 5: ExitPlanMode로 승인 요청

표준 Plan Mode 흐름대로 `ExitPlanMode` 를 호출한다.

- **승인됨**: Step 6(Notion 저장)으로 진행한다. **코드 구현을 시작하지 않는다.**
- **거절됨**: Plan Mode를 닫고 일반 대화(논의)로 돌아간다. 저장하지 않는다.

> Plan Mode 승인은 "계획서 내용에 동의함"을 의미한다. 구현 시작 신호가 아니다.
> 구현은 사용자가 `/clear` 후 새 세션에서 별도로 지시한다.

---

### Step 6: notion-report 스킬에 위임

승인 직후 `notion-report` 스킬을 호출하여 Notion Reports DB에 저장한다.

**전달 내용**:
- 유형: `구현계획`
- 제목: 계획서 제목
- 본문: Plan Mode에서 작성한 계획서 전문 (형식 그대로)
- 프로젝트: 이슈/워킹 디렉터리/대화 맥락에서 파악한 프로젝트명

---

## 주의사항

- **구현하지 않는다.** 코드 파일을 생성·수정하지 않는다.
- 이슈 본문에 모호한 부분이 있으면 임의 결정하지 않고 반드시 사용자에게 확인한다.
- 코드베이스 분석 중 이슈 내용과 괴리(없는 파일·심볼 등)가 발견되면 즉시 피드백한다.
- ExitPlanMode 승인이 없으면 Notion에 저장하지 않는다.
- 브랜치명은 `git-convention` 스킬 컨벤션을 따른다.
- 커밋 메시지 작성은 `syai-commit` 스킬이 담당한다. 계획서의 커밋 계획 섹션에는 메시지 형식이 아닌 작업 단위(무엇을 나눌지)만 기술한다.
- 계획서 끝에 `## 코드 리뷰` 섹션을 넣어 "모든 작업·커밋 완료 후 `/code-review`(및 커스텀 리뷰 스킬)를 실행"하도록 **지시만** 적는다. draft-plan이 직접 코드 리뷰를 실행하지 않는다 — 실행은 구현 세션의 몫이다.
- Notion 저장은 `notion-report` 스킬에 위임한다. draft-plan이 직접 Notion API를 호출하지 않는다.
